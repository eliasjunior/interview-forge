# shared (`@mock-interview/shared`)

The single source of truth for all TypeScript types in the monorepo. Every type that crosses a package boundary lives here — no local `types.ts` files allowed.

> **Monorepo:** this package is one of four. See the [root README](../README.md) for the full picture.
> Related: [interview-mcp](../interview-mcp/README.md) · [report-mcp](../report-mcp/README.md) · [ui](../ui/README.md)

---

## Why this exists

Before the monorepo split, `interview-mcp`, `report-mcp`, and `ui` each had their own copy of `types.ts`. They diverged silently. This package replaces all three with a single file that every workspace imports.

Because all imports are `import type { … }`, TypeScript erases them completely at compile time. There is no runtime dependency on this package — no build step, no bundling, just type-checking.

---

## How to import

```typescript
import type { Session, KnowledgeGraph, Evaluation } from '@mock-interview/shared'
```

Use `import type` (not `import`) — these are compile-time types only.

## Type catalogue

### Interview state machine

**`InterviewState`** — union of all valid session states:
`'ASK_QUESTION' | 'WAIT_FOR_ANSWER' | 'EVALUATE_ANSWER' | 'FOLLOW_UP' | 'ENDED'`

### Session data

**`Message`** — a single turn in the interview transcript:
- `role`: `'interviewer' | 'candidate'`
- `content`: the text
- `timestamp`: ISO string

**`Evaluation`** — result of evaluating one question:
- `questionIndex`, `question`, `answer`
- `score`: 1–5
- `feedback`: one actionable sentence
- `needsFollowUp`: true if score ≤ 3
- `followUpQuestion?`: optional follow-up
- `deeperDive?`: markdown bullets generated by AI

**`Concept`** — a concept extracted from the interview transcript:
- `word`: the concept label
- `cluster`: one of `core concepts | practical usage | tradeoffs | best practices`

**`Session`** — the full session record persisted in the shared runtime store:
- `id`, `topic`, `state`, `currentQuestionIndex`
- `questions[]`, `messages[]`, `evaluations[]`
- `summary?`, `concepts?`
- `createdAt`, `endedAt?`
- `knowledgeSource`: `'file' | 'ai'` — where questions came from

### Knowledge graph

**`GraphNode`** — a concept node in the cumulative graph:
- `id`, `label`
- `clusters[]`: all clusters this concept appears in (can be multiple)

Node identity is canonical: wording variants should collapse into one node ID before the graph is persisted.

Example: `spring-mvc` and `Spring MVC` should resolve to the same node identity instead of creating two separate nodes.

Automatically handled variation classes include:
- case differences
- spaces vs hyphens vs underscores
- repeated separator cleanup
- any explicit alias configured in the graph canonicalisation rules

**`GraphEdge`** — a weighted relationship edge:
- `source`, `target`: node IDs
- `weight`: incremented each time the same edge is observed again
- `kind`: `'cooccurrence' | 'semantic'`
- `relation`: relationship label such as `'co-occurs-with'` or `'used-for'`

**`KnowledgeGraph`** — the full graph persisted in the shared runtime store:
- `nodes[]`, `edges[]`
- `sessions[]`: IDs of sessions that contributed to this graph

### AI provider

**`EvaluationResult`** — the structured response from the Worker LLM's `evaluateAnswer` call:
- `score`, `feedback`, `needsFollowUp`, `followUpQuestion?`, `deeperDive?`

### HTTP API

**`ReportMeta`** — summary row returned by `GET /api/reports`:
- `id`, `topic`, `avgScore`, `date`, `file`

## Adding or changing a type

Edit `shared/src/types.ts` directly. Because TypeScript resolves the type from source (no compile step needed), all three packages pick up the change immediately on the next type-check or build.

Do **not** re-add a `types.ts` in any of the other packages.
